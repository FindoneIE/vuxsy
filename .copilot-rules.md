# Copilot Rules — Findone

Follow these rules strictly when generating code for this project.

## Core principle
Findone is a SaaS listings platform with 3 sections:

- services
- requests
- marketplace

All 3 sections must share one common listing engine.

Avoid code duplication.

---

## Routing rules

Use this exact route structure:

- /services
- /services/new
- /services/[category]
- /services/[category]/[listingId]

- /requests
- /requests/new
- /requests/[category]
- /requests/[category]/[listingId]

- /marketplace
- /marketplace/new
- /marketplace/[category]
- /marketplace/[category]/[listingId]

### Important
- Never use `[id]`
- Always use `[listingId]`
- `[category]` must always come before `[listingId]`
- services, requests, and marketplace must stay structurally identical

---

## Location model

Use only:

- county
- area

Do not use:
- town
- towns

Allowed files:

- src/lib/locations/counties.ts
- src/lib/locations/areas.ts
- src/lib/locations/normalizeLocation.ts

Never create `towns.ts`.

---

## Folder responsibilities

### `src/app`
Routes, pages, and API routes only.

Do not put business logic here.

### `src/components`
UI only.

Do not put:
- Firebase initialization
- Firestore queries
- business logic

### `src/lib`
All business logic must go here.

### `src/hooks`
Client hooks only.

### `src/constants`
Static constants only.

---

## Shared listing engine

Use shared components for all listing types.

Shared listing UI goes in:

- `src/components/listings/`

Shared form UI goes in:

- `src/components/forms/`

Shared listing business logic goes in:

- `src/lib/listings/`

Type-specific differences must be handled with:

- `ServiceFields.tsx`
- `RequestFields.tsx`
- `MarketplaceFields.tsx`

Do not create separate duplicated systems for services, requests, and marketplace.

---

## Firebase rules

Findone uses Firebase.

Firebase files must exist only in:

- `src/lib/firebase/firebase.ts`
- `src/lib/firebase/firebaseAdmin.ts`

Do not initialize Firebase in components.

---

## Firebase Auth rules

Findone uses Firebase Authentication.

Auth helpers must exist only in:

- `src/lib/auth/getCurrentUser.ts`
- `src/lib/auth/requireUser.ts`
- `src/lib/auth/getBearerToken.ts`

Do not place auth logic in UI components.

---

## Firebase Storage and image rules

Image helpers must exist only in:

- `src/lib/images/uploadDraftImages.ts`
- `src/lib/images/imagePaths.ts`
- `src/lib/images/imageValidation.ts`

Do not place image upload or storage logic in UI components.

---

## Listing promotions preparation

Architecture must support future:

- spotlight listings
- bump up
- renew listing

Prepare structure only.

Do not implement payments yet.

Listing model must support fields like:

- spotlight
- spotlightUntil
- bumpedAt
- renewedAt

---

## Grid and list view

Listings must support:

- grid view
- list view

Shared components for this:

- `ListingsGrid.tsx`
- `ListingsList.tsx`
- `ListingViewToggle.tsx`
- `ListingsPageShell.tsx`

Do not duplicate grid/list logic per section.

---

## File creation rules

Before creating a file:
1. Check whether similar logic already exists
2. Reuse existing folders
3. Do not invent new architecture

---

## Code generation rules

When generating code:
- keep it minimal
- follow agreed structure strictly
- prefer reusable shared components
- do not invent extra folders
- do not invent alternative routing
- do not use `[id]`

If unsure, keep code aligned with:
- `.copilot-rules.md`
- `PROJECT_STRUCTURE.md`
